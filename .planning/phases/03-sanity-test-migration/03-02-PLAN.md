<!-- Pitfall #6 context from research -->

Ginkgo JUnit XML known issues with Polarion:
1. Missing `<properties>` section at test suite level (Polarion requires project ID here)
2. `<system-out>` elements absent from passed test cases
3. Test case names formatted as hierarchical paths `[Describe] [Context] [It]` instead of flat IDs
4. Polarion JUMP importer expects `polarion-testcase-id` property or test case ID in name

Ginkgo JunitReportConfig options:
- OmitLeafNodeType: removes [It] prefix from test names
- OmitSuiteSetupNodes: hides BeforeSuite/AfterSuite from report
- OmitSpecLabels: removes label info from test names
- OmitTimelinesForSpecState: reduces verbosity

The test case IDs (PIPELINES-XX-TCNN) are embedded in Describe/It description strings,
so they will appear in JUnit XML test names regardless of Ginkgo formatting.
</interfaces>

  <action>
Create `pkg/junit/transform.go` -- a Go utility that post-processes Ginkgo's JUnit XML output to meet Polarion JUMP importer requirements.

The utility must:

1. **Parse** the JUnit XML file produced by `ginkgo --junit-report=report.xml`.

2. **Inject `<properties>` block** at the test suite level with the Polarion project ID:
   ```xml
   <properties>
     <property name="polarion-project-id" value="PIPELINES" />
     <property name="polarion-testrun-title" value="OpenShift Pipelines Ginkgo Sanity" />
   </properties>
   ```
   The project ID should be configurable via function parameter (default: "PIPELINES").

3. **Extract Polarion test case IDs** from test case names. Ginkgo formats names as `[SUITE] Describe text It text`. The Polarion ID pattern is `PIPELINES-\d+-TC\d+`. Extract this and set it as a property on each test case:
   ```xml
   <testcase name="[Versions Suite] PIPELINES-22: Versions PIPELINES-22-TC01: Check server side components versions">
     <properties>
       <property name="polarion-testcase-id" value="PIPELINES-22-TC01" />
     </properties>
   </testcase>
   ```

4. **Write** the transformed XML to an output file (or overwrite in-place with `--in-place` flag).

Implementation approach:
- Use `encoding/xml` for parsing and writing
- Define structs for JUnit XML schema: `TestSuites`, `TestSuite`, `TestCase`, `Property`
- Use `regexp.MustCompile(`PIPELINES-\d+-TC\d+`)` to extract Polarion IDs from test names
- Provide a `TransformForPolarion(inputPath, outputPath, projectID string) error` function
- Also provide a CLI-callable `main` in `cmd/junit-transform/main.go` if desired, but the primary interface is the Go function

The transform is intentionally simple -- it adds properties, it does not restructure the XML. This preserves Ginkgo's output while adding Polarion metadata.

Do NOT use external XML libraries -- `encoding/xml` is sufficient.

  <name>Task 2: Create JUnit validation script and verify dry-run output</name>
  <files>
    scripts/validate-junit.sh

  <action>
Create `scripts/validate-junit.sh` -- a shell script that validates JUnit XML output meets Polarion requirements. This script is used in CI to catch JUnit regressions before Polarion upload.

The script should:

1. Accept a JUnit XML file path as argument: `./scripts/validate-junit.sh report.xml`

2. Check that the file exists and is valid XML (use `xmllint --noout` if available, or simple grep checks).

3. Verify that **every `<testcase>` element** has a name containing a Polarion test case ID matching the pattern `PIPELINES-[0-9]+-TC[0-9]+`. Print the count of test cases with IDs and without IDs.

4. Verify the expected count of test cases. Accept an optional second argument for expected count: `./scripts/validate-junit.sh report.xml 9`

5. Print a summary:
   ```
   JUnit Validation Report
   =======================
   Total test cases: 9
   With Polarion ID: 9
   Without Polarion ID: 0
   Status: PASS
   ```

6. Exit 0 on success, exit 1 on failure.

Make the script executable: `chmod +x scripts/validate-junit.sh`

Then generate a dry-run JUnit XML and validate it:

```bash
# Generate JUnit XML from dry-run (no cluster needed)
cd /path/to/project
ginkgo run --dry-run --junit-report=sanity-dryrun.xml --label-filter=sanity ./tests/versions/ ./tests/pipelines/ 2>/dev/null || true

# If ginkgo CLI is not in PATH, use:
go run github.com/onsi/ginkgo/v2/ginkgo run --dry-run --junit-report=sanity-dryrun.xml --label-filter=sanity ./tests/versions/ ./tests/pipelines/ 2>/dev/null || true

# Validate the output
./scripts/validate-junit.sh sanity-dryrun.xml 9
```

Note: `--dry-run` may produce XML with 0 test cases if the suites have `BeforeSuite` that fails without a cluster connection. In that case, the validation script should handle this gracefully and report what it found. The important thing is that the script EXISTS and is ready for when real cluster tests produce JUnit output.

If dry-run does not produce usable XML (because BeforeSuite fails without a cluster), create a sample JUnit XML file at `testdata/junit/sample-sanity-report.xml` with the expected structure from the 9 sanity tests. Use this sample to test the validation script and the transform utility:

```bash

go run ./pkg/junit/transform.go testdata/junit/sample-sanity-report.xml testdata/junit/sample-polarion-report.xml PIPELINES
# Validate transformed output
./scripts/validate-junit.sh testdata/junit/sample-polarion-report.xml 9
```
