<task type="auto">
  <name>Task 1: Create diagnostic collection package</name>
  <files>pkg/diagnostics/diagnostics.go</files>
  <action>
Create `pkg/diagnostics/diagnostics.go` that provides a reusable diagnostic collector for `ReportAfterEach`. The package must:

1. **Export `CollectOnFailure(namespace string)`** -- a function suitable for use as `ReportAfterEach(diagnostics.CollectOnFailure(ns))`. It should:
   - Check `CurrentSpecReport().State` -- only collect on `SpecStateFailed`, `SpecStatePanicked`, `SpecStateTimedout`, `SpecStateInterrupted`
   - If not a failure state, return immediately (no overhead on passing tests)

2. **Collect three categories of diagnostics** when test fails:
   a. **Events** -- run `oc get events -n {namespace} --sort-by=.lastTimestamp` to get recent cluster events. Cap output at 100 lines.
   b. **Resource state** -- run `oc get pipelineruns,taskruns,pods -n {namespace} -o wide` to show resource status.
   c. **Pod logs** -- for each pod in the namespace, collect the last 50 lines of logs from each container. Use `oc logs -n {namespace} {pod} -c {container} --tail=50`. Limit to 10 pods to prevent excessive collection.

3. **Attach diagnostics via AddReportEntry**:
   ```go
   AddReportEntry("Cluster Events", eventsOutput, ReportEntryVisibilityFailureOrVerbose)
   AddReportEntry("Resource State", resourceOutput, ReportEntryVisibilityFailureOrVerbose)
   AddReportEntry("Pod Logs", podLogsOutput, ReportEntryVisibilityFailureOrVerbose)
   ```

4. **Handle errors gracefully** -- if `oc` commands fail (e.g., namespace already deleted), log the error but do not fail the diagnostic collection. Use `exec.Command` directly rather than pkg/cmd to avoid test framework assertions in the reporter.

5. **Use `exec.CommandContext` with a 30-second timeout** per oc command to prevent diagnostic collection from hanging.

The function signature should be:
```go
func CollectOnFailure(namespace string) func(ginkgo.SpecReport)
```

This returns a closure that captures the namespace and can be registered with `ReportAfterEach`.

Import `os/exec`, `context`, `time`, `strings`, `fmt`, and Ginkgo v2.

<task type="auto">
  <name>Task 2: Create JUnit-to-Polarion post-processing tool</name>
  <files>cmd/junit-polarion/main.go</files>
  <action>
Create `cmd/junit-polarion/main.go` as a standalone Go program that transforms Ginkgo JUnit XML output into Polarion-compatible format. The tool must:

1. **Accept CLI arguments**:
   - `--input` or `-i`: path to Ginkgo JUnit XML file (required)
   - `--output` or `-o`: path for transformed XML output (defaults to overwriting input)
   - `--project-id` or `-p`: Polarion project ID (defaults to `PIPELINES`, overridable)

2. **Parse JUnit XML** using `encoding/xml`. Define structs matching JUnit schema:
   ```go
   type JUnitTestSuites struct {
       XMLName    xml.Name         `xml:"testsuites"`
       TestSuites []JUnitTestSuite `xml:"testsuite"`
   }
   type JUnitTestSuite struct {
       Name       string            `xml:"name,attr"`
       Tests      int               `xml:"tests,attr"`
       Failures   int               `xml:"failures,attr"`
       Errors     int               `xml:"errors,attr"`
       Time       string            `xml:"time,attr"`
       Properties []JUnitProperty   `xml:"properties>property"`
       TestCases  []JUnitTestCase   `xml:"testcase"`
   }
   type JUnitTestCase struct {
       Name      string          `xml:"name,attr"`
       Classname string          `xml:"classname,attr"`
       Time      string          `xml:"time,attr"`
       Failure   *JUnitFailure   `xml:"failure,omitempty"`
       Skipped   *JUnitSkipped   `xml:"skipped,omitempty"`
       SystemOut string          `xml:"system-out,omitempty"`
   }
   type JUnitFailure struct {
       Message string `xml:"message,attr,omitempty"`
       Type    string `xml:"type,attr,omitempty"`
       Body    string `xml:",chardata"`
   }
   type JUnitSkipped struct {
       Message string `xml:"message,attr,omitempty"`
   }
   type JUnitProperty struct {
       Name  string `xml:"name,attr"`
       Value string `xml:"value,attr"`
   }
   ```

3. **Transform for Polarion compatibility**:
   a. Inject `<properties>` at test suite level:
      - `<property name="polarion-project-id" value="{project-id}"/>`
      - `<property name="polarion-testrun-id" value="{suite-name}-{timestamp}"/>`
   b. Extract test case IDs from spec names using regex `(PIPELINES-\d+-TC\d+)` and set as `<property name="polarion-testcase-id" value="{id}"/>` on test cases that have Polarion IDs.
   c. Ensure every test case has a `<system-out>` element (add empty one if missing).

4. **Write transformed XML** with `xml.MarshalIndent` using 2-space indentation and XML header.

5. **Print summary** to stderr: number of test suites processed, total test cases, test cases with Polarion IDs mapped, test cases without IDs (warns about unmapped tests).

Use only standard library packages: `encoding/xml`, `flag`, `fmt`, `os`, `regexp`, `strings`, `time`.
