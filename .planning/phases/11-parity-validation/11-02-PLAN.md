<task type="auto">
  <name>Task 1: Create pass/fail comparison script and Polarion XML validator</name>
  <files>scripts/parity-results.sh, scripts/parity-polarion.sh</files>
  <action>
Two scripts:

1. **scripts/parity-results.sh** -- Runs both suites and produces a delta report:

   The script orchestrates:
   a. Record cluster state: `oc version`, TektonConfig status, operator version
   b. Run Gauge suite with JUnit output:
      ```bash
      cd reference-repo/release-tests
      gauge run --env=default specs/ 2>&1
      # Collect JUnit XML from reports directory
      cp reports/xml-report/result.xml /tmp/gauge-junit.xml
      ```
   c. Reset cluster state between runs:
      - Delete all test namespaces (`oc delete ns -l created-by=release-tests`)
      - Wait for TektonConfig to reconcile (`oc wait tektonconfig/config --for=condition=Ready`)
   d. Run Ginkgo suite with JUnit output:
      ```bash
      ginkgo run --junit-report=ginkgo-junit.xml ./tests/...
      cp ginkgo-junit.xml /tmp/ginkgo-junit.xml
      ```
   e. Parse both JUnit XMLs to extract test name + status:
      - Use xmllint or awk to extract `<testcase name="..." ...>` and check for `<failure>` child elements
      - Normalize test names to Polarion IDs (extract PIPELINES-XX-TCXX)
      - Produce sorted lists: `test-id PASS|FAIL`
   f. Compare the two sorted lists:
      - Matching results (same ID, same status)
      - Divergent results (same ID, different status)
      - Missing results (ID in one but not the other)
   g. Output delta report to stdout and to `/tmp/parity-delta-report.txt`
   h. Exit 0 if zero unjustified divergences, exit 1 otherwise

   Make the script executable. Include comments explaining each step.
   Script should be defensive: check prerequisites (oc logged in, gauge available, ginkgo available) before starting.

2. **scripts/parity-polarion.sh** -- Compares JUnit XML structure for Polarion compatibility:

   The script validates:
   a. Both JUnit XMLs have the same `<testsuite>` structure
   b. Both have matching `<testcase>` entries (by Polarion test case ID)
   c. Ginkgo JUnit XML has required Polarion fields:
      - `testcase[@name]` contains PIPELINES-XX-TCXX pattern
      - `testcase[@classname]` is in expected format
      - `<properties>` section present if required by Polarion
   d. Compare: for each Polarion test case ID, verify same pass/fail recorded in both XMLs
   e. Optionally submit to Polarion JUMP importer if credentials available (with `--submit` flag)
   f. Output comparison report showing: matching entries, format differences, missing entries

   Make the script executable. If Polarion is not accessible, the script should still validate XML structure and report results with a note that Polarion submission requires `--submit` flag with credentials.

Key considerations:
- Ginkgo JUnit XML uses hierarchical test names (Describe > Context > It). Extract Polarion ID with: `grep -oP 'PIPELINES-\d+-TC\d+'`
- Gauge JUnit XML uses flat scenario names. Same Polarion ID extraction.
- Both scripts must handle the case where some tests are skipped (not failed, just not executed).
- Account for DROPPED-TESTS.md: tests in Gauge but not Ginkgo are expected if they appear in the manifest.
